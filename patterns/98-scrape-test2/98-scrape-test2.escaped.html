id="install"></a>Install</h3> <ul> <li>On Mac OS X: <ul> <li>Install<a href="http://brew.sh">Homebrew</a>.</li> </ul> </li> <li>On other Unix-like OSs: <ul> <li>Permissions might need to be reworked in order to globally install NPMs.</li> <li>It is a bad practice to run npm as root.</li> <li>If necessary, recursively chown the global node_modules directory with the standard user&#39;s ownership.</li> </ul> </li> <li>On non-Unix-like OSs: <ul> <li>Sorry, but Fepper is not supported on non-Unix-like OSs.</li> </ul> </li> <li>Install Node.js and NPM (Node Package Manager). <ul> <li>Requires Node.js v4.0.0 at the very least.</li> <li>On a Mac:<code>brew install node</code></li> <li>If already installed, be sure the version is up to date:<code>node -v</code></li> <li>Update if necessary:<code>brew update &amp;&amp; brew upgrade node</code></li> <li>If not on a Mac, and not using Homebrew:<a href="https://nodejs.org/en/download/package-manager/">https://nodejs.org/en/download/package-manager/</a></li> <li>After installing Node,<code>npm install -g fepper-cli</code></li> </ul> </li> <li>On Mac OS X: <ul> <li>Double-click<code>fepper.command</code></li> </ul> </li> <li>On other OSs (or if you prefer the command line): <ul> <li><code>npm install</code></li> </ul> </li> <li>After successful installation: <ul> <li>Double-click<code>fepper.command</code>again</li> <li>Or enter<code>fp</code>on the command line.</li> </ul> </li> <li>Open<a href="http://localhost:3000">http://localhost:3000</a>in a browser if it doesn&#39;t open automatically.</li> <li>Consult the<a href="http://patternlab.io/docs/index.html">Pattern Lab docs</a>for instructions on using Pattern Lab.</li> <li>Start editing files in<code>patternlab-node/source</code>. Changes should automatically appear in the browser. <ul> <li>If changes do not appear immediately, it may be necessary to install a<a href="http://livereload.com/extensions/">LiveReload browser extension</a>.</li> </ul> </li> <li>To halt Fepper, go to the command line where Fepper is running and press Ctrl+c.</li> </ul> <h3><a id="update"></a>Update</h3> <p>The easiest way to update Fepper is with Git. This generally means forking an Electric Eloquence Fepper repository at your project&#39;s onset, and pulling from the master branch of the main upstream repository when updates are released.</p> <p>All updates must pull from<a href="https://github.com/electric-eloquence/fepper">https://github.com/electric-eloquence/fepper</a>even if your original installation was forked from a child project. Pulling from a child project&#39;s repository may overwrite custom code specific to your project so don&#39;t do it.</p> <p>If not updating with Git, you&#39;ll need to copy the latest package over the old. Be sure to take care that &quot;dot&quot;-prefixed hidden files at the root level are<em>not</em>overwritten. These are important configuration files specific to each user&#39;s instance.</p> <h3><a id="configure"></a>Configure</h3> <p>Edit<code>pref.yml</code>to customize preferences and to view further documentation in the comments. If you wish to use the<code>syncback</code>,<code>frontend-copy</code>, or<code>template</code>tasks, you must supply values for the<code>backend.synced_dirs</code>preferences in order for those directories to get processed and copied to the backend.</p> <h3><a id="use"></a>Use</h3> <ul> <li>To launch from the Mac OS X Finder: <ul> <li>Double-click<code>fepper.command</code></li> </ul> </li> <li>To launch from the command line: <ul> <li><code>fp</code></li> </ul> </li> <li>These other utility tasks are runnable on the command line: <ul> <li><code>fp data</code>to force compile data.json.</li> <li><code>fp frontend-copy</code>to copy assets, scripts, and styles to the backend.</li> <li><code>fp lint</code>to lint HTML, JavaScripts, and JSON.</li> <li><code>fp minify</code>to minify JavaScripts.</li> <li><code>fp once</code>to clean the public folder and do a one-off Fepper build.</li> <li><code>fp publish</code>to publish the public folder to GitHub Pages.</li> <li><code>fp restart</code>to restart after shutdown, but without opening the browser.</li> <li><code>fp static</code>to generate a static site from the 04-pages directory.</li> <li><code>fp syncback</code>combines lint, minify, frontend-copy, and template.</li> <li><code>fp template</code>translates templates for the backend and copies them there.</li> </ul> </li> <li>If using Git for version control, directories named &quot;ignore&quot; will be ignored.</li> </ul> <h3><a id="global-data"></a>Global Data</h3> <p>Edit<code>patternlab-node/source/_data/_data.json</code>to globally populate Mustache templates with data. Manual edits to<code>patternlab-node/source/_data/data.json</code>will get overwritten on compilation.</p> <h3><a id="partial-data"></a>Partial Data</h3> <p>Underscore-prefixed .json files within<code>patternlab-node/source/_patterns</code>will be concatenated to the output of<code>_data.json</code>, the whole in turn getting compiled into<code>data.json</code>, the final source of globally scoped data.</p> <p><em>Partial data</em>is distinct from<em>pattern data</em>. For example,<code>00-homepage.json</code>is<em>pattern data</em>and specific to the<code>00-homepage</code>pattern. No other pattern will pick up<code>00-homepage.json</code>, even if<code>00-homepage.mustache</code>is included in another pattern. However,<code>_00-homepage.json</code>is<em>partial data</em>and will get concatenated to the<em>global data</em>outputted to<code>data.json</code>.<code>_00-homepage.json</code>will be picked up by all patterns.</p> <ul> <li><strong>DO NOT EDIT patternlab-node/source/_data/data.json</strong></li> <li><strong>DO PUT GLOBAL DATA IN patternlab-node/source/_data/_data.json</strong></li> <li><strong>DO LIBERALLY USE PARTIAL DATA IN patternlab-node/source/_patterns FOR ORGANIZATIONAL SANITY</strong></li> </ul> <h3><a id="static-site-generation"></a>Static Site Generation</h3> <p>Running<code>fp static</code>will generate a complete static site based on the files in<code>patternlab-node/source/_patterns/04-pages</code>. The site will be viewable at<a href="http://localhost:3000/static/">http://localhost:3000/static/</a>. An<code>index.html</code>will be generated based on<code>04-pages-00-homepage</code>or whatever is defined as the homepage in<code>_data.json</code>. If the links are relative and they work correctly in the Pattern Lab UI, they will work correctly in the static site even if the<code>static</code>directory is moved and renamed. The only caveat is that links to other pages in the<code>patterns</code>directory must start with<code>../04-pages-</code>and not<code>../../patterns/04-pages-</code>.</p> <h3><a id="the-backend"></a>The Backend</h3> <p>Fepper can almost as easily work with a CMS backend such as Drupal or WordPress, while not requiring Apache, MySQL, or PHP. Put the actual backend codebase or even just a symbolic link to the codebase into the<code>backend</code>directory. Then, enter the relative paths to the appropriate backend directories into<code>pref.yml</code>. (Do not include &quot;backend&quot; or a leading slash.) You will then be able to run<code>fp syncback</code>or<code>fp frontend-copy</code>to export your frontend data into your backend web application.</p> <ul> <li>Be sure that<code>backend.synced_dirs.assets_dir</code>,<code>backend.synced_dirs.scripts_dir</code>, and<code>backend.synced_dirs.styles_dir</code>are set in<code>pref.yml</code>.</li> <li>The above values set in<code>pref.yml</code>can be overridden on a per-file basis by similarly named YAML files with similarly named settings.</li> <li>These YAML files must match the source file&#39;s name with exception of the extension.</li> <li>The extension must be<code>.yml</code></li> <li>The overriding property must only contain the lowest level key-value, not the entire hierarchy, i.e. only<code>assets_dir</code>,<code>scripts_dir</code>, or<code>styles_dir</code> </li> <li>Files prefixed by &quot;__&quot; will be ignored as will files in the<code>_nosync</code>directory at the root of the source directories.</li> </ul> <h3><a id="templater"></a>Templater</h3> <p>Pattern Lab&#39;s Mustache templates can be translated into templates compatible with your backend. Mustache tags just need to be replaced with tags the backend can use. Put these translations into YAML files named similarly to the Mustache files in<code>patternlab-node/source/_patterns/03-templates</code>. Follow the example in<code>test/files/_patterns/03-templates/00-homepage.yml</code>for the correct YAML syntax.</p> <p>Follow these rules for setting up keys and values:</p> <ul> <li>Delete the outer two curly Mustache braces for keys.</li> <li>Trim any exterior whitespace.</li> <li>Leave other control structures within the key, i.e., !#/&gt;^{}</li> <li>Escape parentheses and question marks with two backslashes.</li> <li>Wrap the key in double quotes.</li> <li>Follow the closing quote with a colon, space, and pipe.</li> <li>Indent each line of the value by at least two spaces.</li> </ul> <p>Run<code>fp syncback</code>or<code>fp template</code>to execute the Templater.</p> <ul> <li>Be sure that<code>backend.synced_dirs.templates_dir</code>and<code>backend.synced_dirs.templates_ext</code>are set in<code>pref.yml</code>.</li> <li>The default<code>templates_dir</code>and<code>templates_ext</code>settings in<code>pref.yml</code>can be overridden by similarly named settings in the template-specific YAML files.</li> <li>Templates prefixed by &quot;__&quot; will be ignored by the Templater as will files in the<code>_nosync</code>directory.</li> <li>The Templater will recurse through nested Mustache templates if the tags are written in the verbose syntax and have the<code>.mustache</code>extension, i.e.<code>{{&gt; 02-organisms/00-global/00-header.mustache }}</code>.</li> <li>However, the more common inclusion use-case is to leave off the extension, and not recurse.</li> </ul> <p><a href="https://github.com/electric-eloquence/fepper-drupal">Fepper for Drupal</a>and<a href="https://github.com/electric-eloquence/fepper-wordpress">Fepper for WordPress</a>have working examples of templates compatible with the Templater.</p> <h3><a id="webserved-directories"></a>Webserved Directories</h3> <p>When using a backend, assets generally need to be shared with the Fepper frontend. The<code>syncback</code>and<code>frontend-copy</code>tasks copy files from Fepper to the backend, but not the other way. Instead of providing a task to copy in the reverse direction, Fepper serves backend files if their directories are entered into the<code>webserved_dirs</code>block in<code>pref.yml</code>. Be sure these values are formatted as YAML array elements.</p> <p><code>DO NOT INCLUDE DIRECTORIES WITH SOURCE CODE! GITHUB PAGES AND MANY OTHER PUBLIC HOSTS DO NOT PREPROCESS PHP AND OTHER PROGRAMMING LANGUAGES, SO ANY PUBLISHED SOURCE CODE WILL BE RENDERED AS PLAIN TEXT! THIS WILL MAKE PUBLIC ANY SENSITIVE INFORMATION CONTAINED WITHIN THE SOURCE CODE!</code></p> <h3><a id="github-pages"></a>GitHub Pages</h3> <p>If you have checked your Fepper instance into a repository in your GitHub account, you may run<code>fp publish</code>to publish<code>patternlab-node/public</code>to GitHub Pages. The Pattern Lab UI and Fepper static files will then be viewable from the Web at<code>http://{user}.github.io/{repo}</code>. Normally, this is all that is needed. However, if you are using<code>webserved_dirs</code>, you will need to supply a<code>gh_pages_prefix</code>preference in<code>pref.yml</code>. This preference needs to be set to the name of your GitHub repository.</p> <h3><a id="mustache-browser"></a>Mustache Browser</h3> <p>Mustache code can be viewed in the Pattern Lab UI by clicking the eyeball icon in the upper right, then clicking Code, and then clicking the Mustache tab in the bottom pane. The Mustache tags are hot-linked, and if they are written in the verbose syntax, clicking on them will open that Mustache file and display its code in the Pattern Lab UI, with its Mustache tags hot-linked as well. The Mustache tags must be coded in the verbose-pathed manner:<code>{{&gt; 02-organisms/00-global/00-header }}</code></p> <p>The path must be correct; however, the<code>.mustache</code>extension is optional. The default homepage is a working example.</p> <h3><a id="html-scraper"></a>HTML Scraper</h3> <p>Fepper can scrape and import Mustache templates and JSON data files from actual web pages. A common use-case is to scrape pages from a backend populated with CMS content in order to auto-generate data, and to replicate the HTML structure. To open the Scraper, click Scrape in the Pattern Lab UI, and then click HTML Scraper. Enter the URL of the page you wish to scrape. Then, enter the CSS selector you wish to target (prepended with &quot;#&quot; for IDs and &quot;.&quot; for classes). Classnames and tagnames may be appended with array index notation ([n]). Otherwise, the Scraper will scrape all elements of that class or tag sequentially. Such a loosely targeted scrape will save many of the targeted fields to the JSON file, but will only save the first instance of the target to a Mustache template.</p> <p>Upon submission, you should be able to review the scraped output on the subsequent page. If the output looks correct, enter a filename and submit again. The Scraper will save Mustache and JSON files by that name in the 98-scrape directory, also viewable under the Scrape menu of the toolbar. The Scraper will correctly indent the Mustache code. However, this beautification may also render some attributes of the Mustache code a little different than the original HTML.</p> <h3><a id="variables.styl"></a>variables.styl</h3> <p><code>patternlab-node/source/_scripts/src/variables.styl</code>is a file containing variables that can be shared across the Stylus CSS preprocessor, browser JavaScripts, and PHP backends (and possibly other language backends as well). It ships with these values:</p> <p><code>bp_lg_max = -1 bp_lg_min = 1024 bp_md_min = 768 bp_sm_min = 480 bp_xs_min = 0</code></p> <p>It cannot contain comments, semi-colons, curly braces, etc. It is straightforward to import and use these variables in Stylus and JavaScript. PHP must import them with<code>parse_ini_file()</code>. Fepper tries to be agnostic about CSS processors and tries to keep the amount of NPMs to download to a minimum, so it does not ship with Stylus (or any other CSS pre/post-processor) configured. However, since Stylus allows for this easy sharing of variables, Fepper does ship with a<code>patternlab-node/source/css-processors/stylus</code>directory which can be compiled into the stock Pattern Lab CSS by configuring<code>extend/custom/css-process/css-process~extend.js</code>. The Stylus files are written in the terse, Python-like, indentation-based syntax; however, the more verbose, CSS-like syntax (with curly braces, colons, and semi-colons) is perfectly valid as well.</p> <h3><a id="more-documentation"></a>More Documentation</h3> <ul> <li><a href="https://github.com/electric-eloquence/fepper/blob/master/excludes/default.pref.yml">default.pref.yml</a></li> <li><a href="http://patternlab.io/docs/index.html">Pattern Lab</a></li> <li><a href="https://mustache.github.io/mustache.5.html">Mustache</a></li> </ul> <h3><a id="contributing"></a>Contributing</h3> <p>Contributions and bug fixes are greatly appreciated!</p> <ul> <li>Please pull request against the<a href="https://github.com/electric-eloquence/fepper/tree/dev">dev branch</a>.</li> <li>Please try to be both concise as well as clear on what is trying to be accomplished.</li> </ul> </main>
